# 主流 AI 大模型 Vibe Coding 适配指南

在 Vibe Coding 的心法中，“契约”是通用的，但由于各个大模型（Claude、OpenAI、DeepSeek）的**底层注意力机制、推理逻辑和产品载体**不同，我们在投喂 Rules（规则）和 Context（上下文）时，需要做针对性的适配。

---

## 一、 Claude Code (Anthropic) 的法则

Claude 系列（尤其是 Claude 3.5/3.7 Sonnet）以其**超大上下文窗口（200K）**和**严谨的代码重构能力**著称。Claude Code 官方 CLI 进一步将其封装为强大的终端工具。

### 1. 偏好：XML 标签包裹与长文分析
Claude 极度偏好 XML 风格的结构化提示词。在编写给 Claude 的规则或提问时，使用标签包裹能极大提升其精准度。

* **适配 Rule**：在你的指令或项目中引入明确的 XML 边界：
  ```xml
  <instructions>
    1. 你必须严格遵循 <schemas> 中定义的数据结构。
    2. 如果发生报错，请按 <error_handling> 的标准进行包装。
  </instructions>
  <schemas>
    (这里贴入你的 schemas.py 核心代码)
  </schemas>
  ```

### 2. 弱点：过度谨慎与“教书匠”属性
Claude 经常喜欢在你要求改代码前，长篇大论地给你解释原理。
* **专属 Rule**：
  `"NO_PREAMBLE: 直接输出代码或执行文件修改。禁止在修改前输出任何前置解释、道歉或“好的，我现在就去改”这类废话。"`

### 3. Claude Code CLI 专属：利用其强大的 Bash 能力
Claude Code 非常擅长自己在终端跑命令找问题。
* **专属 Rule**：
  `"当遇到测试失败时，不要立刻盲目修改代码。必须先自行运行 pytest 分析报错栈，或者使用 grep 搜索相关文件，再提出修改方案。"`

## 二、 OpenAI (GPT-4o) / ChatGPT 的法则

GPT-4o 是一个极其聪明的“多面手”，但其代码上下文窗口相对较小（相比 Claude），且在处理超长代码时容易出现“偷懒”（例如使用 `// ... remaining code`）。

### 1. 偏好：清晰的 Markdown 结构与 Few-Shot（少样本学习）
GPT-4o 对 Markdown 的解析极佳。它非常依赖“你给它看一个好例子，它就会照猫画虎”。

* **适配 Rule**：
  ```markdown
  ### 核心指令
  请实现文件解析功能。

  ### 黄金示例 (Few-Shot)
  **输入：** `[a, b, c]`
  **期望风格：**
  ```python
  def parse(data: list) -> list:
      return [item.upper() for item in data if item]
  ```

### 2. 弱点：“偷懒”占位符
GPT-4o 在重构大文件时，极其喜欢用注释代替代码，导致代码被破坏。
* **专属 Rule（必须加粗大写）**：
  `"DO NOT USE PLACEHOLDERS. 无论文件多长，必须输出完整、未删减、可直接复制运行的完整代码。绝对禁止使用 '// ...' 或 'rest of the code' 等省略占位符！"`

### 3. 弱点：幻觉依赖
GPT-4o 喜欢自己发明库或者使用过时的 API。
* **专属 Rule**：
  `"严格依赖锁定：如果你发现需要使用第三方库，请先通过 shell 检查当前虚拟环境中是否已安装。未安装的库，必须先向我申请许可才能编写代码。"`

## 三、 DeepSeek (V3/R1) 的法则

DeepSeek，特别是以强化学习驱动的 R1 模型（具有深度思考 `<think>` 过程），在算法逻辑和深层 Bug 排查上展现出惊人的能力。但其工具调用（Tool Use）和工程工程规范性可能不如前两者圆滑。

### 1. 偏好：逻辑链拆解与“思维推演”
与其直接告诉 DeepSeek 最终的规则，不如让它看到规则背后的“为什么”。R1 模型如果在 `<think>` 阶段没有理顺逻辑，输出就会走偏。

* **适配 Rule**：不要只给死命令，给出推理前缀。
  ```text
  【分析要求】：在动手写代码前，你必须先在思考过程中完成以下自查：
  1. 当前请求的数据结构是否符合 Pydantic Schema 的要求？
  2. 如果请求发生网络超时，当前的重试机制能否捕获？
  只有这两个问题你想清楚了，再输出最终代码。
  ```

### 2. 弱点：工程文件的“过度发散”
DeepSeek 在写代码时，有时会为了炫技，引入复杂的算法或脱离工程实际的设计模式。
* **专属 Rule**：
  `"KISS (Keep It Simple, Stupid) 原则：严禁过度设计。请使用最平庸、最易读的 Python 原生语法。不要使用元类（metaclass）或复杂的闭包。完全照搬 examples/ 目录下的平庸实现。"`

### 3. 强项：算法与底层查错
如果项目里遇到了极其诡异的并发问题、死锁或者性能瓶颈，DeepSeek 是最强的。
* **专属 Prompt 模板**：
  `"这里有一段存在竞争条件的代码和报错日志。请开启深度推理模式，逐行追踪变量在不同线程下的状态变化，找出死锁的根因，并给出修复建议。修复必须遵循不增加新锁的原则。"`

---

## 总结：看人下菜碟

* **用 Claude (特别是配合 CLI)**：把项目丢给它，用 **XML** 定好 Schema，让它自己去跑测试、改代码。它是最强的自动驾驶仪。
* **用 OpenAI (GPT-4o)**：适合做从 0 到 1 的起步。给它 **Markdown** 和 **Few-Shot 例子**，严格禁止它偷懒省略代码。
* **用 DeepSeek**：遇到难啃的算法骨头或诡异 Bug。用**逻辑引导**触发它的深度思考，但要用 **KISS 原则**防止它写出无法维护的“天书”。
