# Gemini CLI 规则设计指南 (Rules Design Guide)

## 一、 什么是 Gemini 的 Rules？（四层契约体系）

Gemini CLI 并不依赖单一的规则文件，而是将规则分为四个维度，按优先级和作用域生效：

1. **全局个人宪法 (`~/.gemini/GEMINI.md`)**
   * **作用域**：跨越你所有的项目。只要是你用 Gemini 开发，它都生效。
   * **写什么**：你的“开发者 DNA”。比如：“我永远偏好异步 Python”、“写前端只用 Vanilla CSS，绝不用 Tailwind”、“变量命名必须是严格的蛇形命名法”。

2. **项目级公约 (`./GEMINI.md` 在项目根目录)**
   * **作用域**：当前仓库。需要被 `git commit` 并与团队共享。
   * **写什么**：项目的核心技术栈、架构模式、PR 规范等（这也是本 Showcase 项目根目录下 `GEMINI.md` 的定位）。

3. **子目录微观法则 (`./src/GEMINI.md`)**
   * **作用域**：仅针对特定子目录。
   * **写什么**：比如在 `frontend/` 下写前端特有规则，在 `backend/` 下写后端规则。它能有效防止上下文污染。

4. **私人工作区记忆 (`~/.gemini/tmp/.../memory/MEMORY.md`)**
   * **作用域**：仅限你当前电脑的当前项目，**绝不提交**。
   * **写什么**：你的本地测试数据库密码、你机器特有的本地路径、或者你正在 debug 的临时思路记录。

## 二、 如何“优雅”地设计项目级 GEMINI.md？

优雅的 Rule 设计，本质上是**“Prompt Engineering（提示词工程）的代码化”**。遵循以下四个原则，你的规则将拥有极高的约束力：

### 1. 负向约束优先（Negative Constraints）
AI 很容易“过度热心”，帮你引入多余的库或做无谓的重构。告诉它“不能做什么”，比告诉它“做什么”更有效。
* ❌ **糟糕的写法**：“请尽量保持类型安全。”
* ✅ **优雅的写法**：“**NEVER bypass the type system.** 严禁使用 `# type: ignore`、`Any` 或隐式类型转换。”

### 2. 建立索引与锚点（Context Anchors）
不要在 Rule 里写长篇大论的业务逻辑，而是把它当成一个“指针”，指向现有的 Golden Examples 或 Schema。
* ❌ **糟糕的写法**：“处理请求时，先校验长度，再发网络请求，如果遇到 HTTP 错误要捕获，并封装成 DataFetchError……”
* ✅ **优雅的写法**：“所有外部 API 请求的实现，**强制对齐** `examples/golden_service.py` 中的防御性编程风格。”

### 3. 剥夺“自主决断权”（Strict Boundaries）
在关键节点上，明确剥夺 AI 擅自做主的权利，强制它向用户确认或遵循特定流程。
* ❌ **糟糕的写法**：“如果遇到错误，请妥善处理。”
* ✅ **优雅的写法**：“在修改现有业务逻辑前，必须先运行 `pytest` 确认测试覆盖情况。如果尝试修复 Bug 超过 3 次依然失败，必须**停止编码**，向用户报告并重新评估架构。”

### 4. 高度结构化的排版
AI 解析 Markdown 的结构能力极强。使用清晰的标题、列表和加粗，让其注意力瞬间聚焦。
